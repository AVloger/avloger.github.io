很多人第一次听到“虚拟内存”，会把它理解成“内存不够时，把磁盘拿来当内存”。

这只说到了 swap 这一小部分，而且容易把重点带偏。

虚拟内存更本质的作用，是操作系统给每个进程提供一套独立的虚拟地址空间。程序访问的是虚拟地址，CPU 通过 MMU 和页表把虚拟地址翻译成物理地址。这个机制让操作系统可以实现进程隔离、权限保护、按需加载、共享库映射、文件映射和 swap。

一句话概括：

> 虚拟内存不是单纯的“扩容内存”，而是现代操作系统管理地址空间和物理内存的核心抽象。

## 1. 什么是虚拟内存

虚拟内存是操作系统对物理内存的一层抽象。

每个进程看到的是自己的、独立的、连续的虚拟地址空间。程序里的指针地址是虚拟地址，不是直接的物理地址。CPU 访问内存时，会通过 MMU 查询页表，把虚拟地址转换成物理地址。

这个抽象带来几个关键能力：

- 每个进程拥有独立地址空间。
- 进程之间默认不能直接访问彼此内存。
- 页表项可以记录读、写、执行权限。
- 程序和动态库可以按需加载。
- 多个进程可以共享同一份物理页。
- 文件可以通过 `mmap` 映射到地址空间。
- 物理内存紧张时，可以把部分页面换出到磁盘。

所以，理解虚拟内存要抓住三个词：

```text
虚拟地址空间
页表映射
隔离与保护
```

## 2. 虚拟地址如何转换成物理地址

现代操作系统通常以页为单位管理内存。

虚拟地址可以拆成：

```text
虚拟地址 = 虚拟页号 + 页内偏移
```

物理地址可以拆成：

```text
物理地址 = 物理页框号 + 页内偏移
```

转换过程大致如下：

1. CPU 生成虚拟地址。
2. MMU 根据虚拟页号查询页表。
3. 页表中找到对应的物理页框号。
4. 物理页框号加上页内偏移，得到物理地址。
5. CPU 访问真实物理内存。

页内偏移不需要转换。因为虚拟页和物理页框大小相同，虚拟地址页内偏移是多少，转换后的物理地址页内偏移也是多少。

如果页表中没有有效映射，就会触发缺页异常。

---

## 3. 页表是什么

页表是“虚拟页到物理页框”的映射表。

页表项通常不只保存物理页框号，还会保存各种状态位和权限位：

| 字段 | 作用 |
|------|------|
| 物理页框号 | 虚拟页实际映射到哪块物理内存 |
| present bit | 页面是否在物理内存中 |
| read/write | 页面是否可写 |
| user/supervisor | 用户态是否可以访问 |
| execute bit | 页面是否可执行 |
| dirty bit | 页面是否被修改 |
| accessed bit | 页面是否被访问过 |

为什么需要多级页表？

因为虚拟地址空间很大。如果为整个虚拟地址空间建立一张完整连续的大页表，会非常浪费。多级页表可以只为实际使用的地址范围分配页表页。

这也是虚拟内存的一个重要思想：地址空间可以看起来很大，但底层资源按需分配。

---

## 4. TLB：地址翻译结果的缓存

如果每次内存访问都要走多级页表，成本会非常高。

TLB，全称 Translation Lookaside Buffer，是 CPU 中用于缓存地址翻译结果的高速缓存。它缓存的不是内存数据，而是虚拟页到物理页框的映射。

访问过程可以简化成：

```text
先查 TLB
命中：直接得到物理页框号
未命中：查页表，再把结果写入 TLB
```

![TLB 地址翻译流程](/images/virtual-memory/tlb.svg)

_图片来源：Wikimedia Commons，TLB.svg，CC BY-SA 4.0。_

TLB 命中率对性能很重要。进程切换、地址空间变化、页表更新，都可能导致 TLB 中的部分映射失效。

大页也和 TLB 有关：同样大小的 TLB，如果每个表项覆盖的内存范围更大，就能减少 TLB miss 的概率。

## 5. 缺页异常是什么

当进程访问一个虚拟页，但页表中没有有效物理页映射时，CPU 会触发缺页异常。

常见原因包括：

- 页面还没有按需加载。
- 页面已经被换出到磁盘。
- 访问 `mmap` 文件中尚未加载的页。
- 访问未映射地址。
- 写入 copy-on-write 页面。
- 越权访问只读或不可执行页面。

内核处理流程大致是：

1. CPU 发现页表项无效或权限不满足，触发异常。
2. 内核接管，判断这次访问是否合法。
3. 如果合法，分配物理页或从磁盘读取页面。
4. 更新页表。
5. 恢复用户态程序继续执行。

如果访问非法，例如访问空指针附近地址、写只读页面、执行不可执行页面，内核就不会为它补映射，程序通常会收到段错误。

## 6. 虚拟内存的核心作用

### 6.1 进程隔离

每个进程有自己的虚拟地址空间。

两个进程可以都认为自己在访问 `0x400000`，但这个虚拟地址映射到的物理地址可能完全不同。

### 6.2 内存保护

页表项里有权限位。操作系统可以把代码段标记为只读可执行，把数据段标记为可读写，把用户态不该访问的内核区域保护起来。

这也是很多安全机制的基础。例如栈不可执行可以降低某些攻击方式的风险。

### 6.3 按需加载

程序启动时，不需要把整个可执行文件和所有动态库一次性加载进物理内存。

用到哪个页面，再加载哪个页面。

### 6.4 共享映射

多个进程可以映射同一份物理页。例如共享库的代码段通常可以被多个进程共享，从而节省内存。

### 6.5 文件映射

`mmap` 可以把文件映射到进程地址空间。程序像访问内存一样访问文件内容，由操作系统负责页面加载和回写。

### 6.6 Swap

当物理内存紧张时，操作系统可以把不活跃页面换出到磁盘。

Swap 是虚拟内存体系的一部分，但不是虚拟内存的全部。

## 7. malloc 和虚拟内存的关系

`malloc` 分配的是进程虚拟地址空间中的一段可用区域，不一定立刻对应真实物理内存。

常见路径是：

- 小块内存：内存分配器从已有 heap arena 中切分。
- 大块内存：可能通过 `mmap` 向操作系统申请。
- 物理页：通常在第一次实际访问时，才由缺页机制分配。

所以，`malloc` 成功更多表示虚拟地址空间和分配器元数据层面分配成功。物理内存可能是按需分配的，真正写入时才触发页面分配。

这也是为什么一些系统会出现 overcommit：程序申请到虚拟内存，并不等于系统已经为它准备好了同等大小的物理内存。

---

## 8. mmap 怎么理解

`mmap` 是把文件或匿名内存映射到进程虚拟地址空间。

常见用途包括：

- 读取大文件，避免手动 `read` 循环。
- 多进程共享内存。
- 分配大块匿名内存。
- 数据库、搜索引擎、日志系统中用于文件缓存和持久化访问。

它的优点是：

- 可以按需分页加载。
- 可以减少用户态和内核态之间的数据拷贝。
- 访问方式像普通内存。

但 `mmap` 不是免费午餐：

- 随机访问可能导致大量缺页。
- 文件被截断后继续访问映射区域可能出错。
- 回写时机需要理解 `msync` 或内核刷脏页策略。
- 错误处理方式和普通 `read` 不一样。

---

## 9. Copy-on-write 为什么能省内存

`fork` 时，父子进程一开始可以共享相同的物理页。

内核会把这些页面标记成只读。只要父子进程都只是读取，就不需要复制物理内存。

当某一方尝试写入时，会触发缺页异常。内核发现这是 copy-on-write 页面，就复制一份新的物理页给写入方，然后更新页表。

这样可以显著降低 `fork` 的内存复制成本。

---

## 10. 几个容易混淆的缓存

| 概念 | 缓存的是什么 |
|------|--------------|
| CPU cache | 内存数据 |
| TLB | 地址翻译结果 |
| page cache | 文件内容 |
| swap | 被换出的匿名页 |
| mmap | 文件或匿名页到虚拟地址空间的映射方式 |

不要把 TLB 说成缓存数据，它缓存的是地址映射。

也不要把 page cache 和 swap 混在一起。Page cache 主要缓存文件内容；swap 主要承载被换出的匿名页。

---

## 11. 一个简洁总结

虚拟内存可以按这条链路理解：

```text
进程看到虚拟地址空间
-> CPU 生成虚拟地址
-> TLB 优先缓存地址翻译结果
-> TLB miss 时查询页表
-> 页表给出物理页框和权限
-> 映射不存在或权限不满足时触发缺页异常
-> 内核决定是补页、加载文件、处理 COW，还是杀掉进程
```

它的核心价值不是“磁盘当内存”，而是让操作系统能用统一机制管理地址空间、权限、隔离、加载、共享和回收。

理解这一点之后，`malloc`、`mmap`、copy-on-write、page cache、swap、段错误这些概念就能串成一张图，而不是一堆零散术语。
